# AutoDCF v3

AutoDCF builds institutional-quality DCF valuation models compliant with institutional Financial Modeling Guidelines. The MAIN CONVERSATION is a pure orchestrator — it spawns separate builder and auditor agents per step. It NEVER writes build code itself.

## Platform Support

This skill works on multiple AI coding assistants. Load the orchestration adapter for your platform:

| Platform | Adapter | Agent Dir |
|----------|---------|-----------|
| **Claude Code** | `platform-adapters/claude-code-orchestrator.md` | `agents/claude-code/` |

## Architecture Overview

```
Terminal (Orchestrator)
  ├── Phase 1: Data Acquisition (7 sub-steps)
  │   └── Spawns: autodcf-data-researcher per sub-step
  ├── Phase 2: Model Build (18 steps)
  │   └── For EACH step:
  │       ├── Spawns: autodcf-builder (writes module, runs orchestrator)
  │       ├── Spawns: dcf-auditor (checklist audit)
  │       ├── Spawns: autodcf-adversarial (free-form hunt)
  │       └── If adversarial finds issues: spawns dcf-auditor (verification)
  └── Phase 3: Integration Audit
      └── Spawns: dcf-auditor + autodcf-adversarial
```

**Excel workbook (4 tabs, institutional standard order):**

| Tab | Purpose |
|-----|---------|
| **Cover** | Model metadata, 7 automated checks with conditional formatting |
| **Outputs** | Two dashboards (Perpetuity + Multiple), each with 4 sensitivity tables = 8 total, cumulative scenario comparison |
| **Inputs** | Drivers (3-scenario CHOOSE switch at F7), WACC (5 real peers, Hamada, CAPM), Other Inputs |
| **Model** | 12 schedules stacked vertically (~320 rows) |

**Column layout** (all data tabs): A=spacer(2.5), B=nav(4), C=labels(35), D=units(8), E=prior(14), F-I=historical 4yr, J-N=forecast 5yr.

**The 12 Model schedules:** Revenue, Cost, Income Statement, Working Capital, Depreciation, Asset (PP&E), Debt, Levered Tax, Unlevered Tax, UFCF (dual method), DCF Perpetuity, DCF Multiple.

## Non-Negotiable Rules

1. **Zero hardcoded data in Excel** — Every value from CSV with full provenance. Every blue cell comments reference CSV row + source doc + page + quote.
2. **Formulas over hardcodes** — Every projection cell is a live Excel formula referencing Inputs.
3. **Font colors** — Blue (#0000CC) = CSV inputs, Black (#000000) = formulas, Green (#006100) = cross-sheet refs.
4. **Named row constants** — Python modules import from row_map.py. NEVER literal row numbers.
5. **Forecast columns only** — Formula loops iterate FCST_COLS. Historical = sacrosanct GAAP actuals from CSV.
6. **Sheet prefix required** — Every cross-sheet formula: `=Model!N138`, never `=N138`.
7. **Driver switch cascades** — Inputs!F7 (1/2/3) drives CHOOSE on ALL driver rows to per-share value.
8. **End-of-year TV** — Discrete FCFs mid-year. Terminal value end-of-year integer period.
9. **Re-discount sensitivity** — Each sensitivity cell re-discounts every FCF at scenario WACC. No linear scaling.
10. **8 sensitivity tables** — 4 per dashboard (WACC vs TGR, WACC vs Multiple, EV-only, Equity Value). Plus cumulative scenario comparison.
11. **Dual UFCF with reconciliation** — EBITDA method + Net Income method + reconciliation = 0.
12. **Double borders on totals** — Every total gets double-bottom. Section headers get single bottom.

## Workflow

### Phase 1: Data Acquisition (Sub-Steps 0a-0g)

For EACH sub-step, spawn a data researcher agent:

```
Agent(subagent_type="autodcf-data-researcher", prompt="""
DATA ACQUISITION SUB-STEP 0{X}: {NAME}

## Your Spec
Read this file for detailed instructions:
{absolute path to references/steps/step_00{X}_{name}.md}

## Company
Ticker: {TICKER}, Exchange: {EXCHANGE}

## Output Paths
CSV: {absolute path to data/{csv_name}.csv}
Artifacts: {absolute path to data/artifacts/}
Report: {absolute path to {TICKER}_reports/step_00{X}_data_report.md}

## Prior Sub-Step Outputs (if any)
{paths to CSVs from prior sub-steps}
""")
```

Sub-steps 0a through 0f run in the order specified (some can be parallelized: 0d+0e can run while 0b+0c run). Sub-step 0g (validation gate) runs LAST and is BLOCKING — build cannot start until 0g passes.

After 0g passes, update BUILD_STATE.md and commit all data/ files.

#### D. SPAWN ADVERSARIAL AUDITOR
```
Agent(subagent_type="autodcf-adversarial", prompt="""
ADVERSARIAL AUDIT STEP {N}: {STEP_NAME}

## Checklist Audit Report (already passed — do NOT repeat these checks)
{absolute path to {TICKER}_reports/step_{NN}_checklist_audit.md}

## Build Report
{absolute path to {TICKER}_reports/step_{NN}_build_report.md}

## Files to Inspect
- Excel: {absolute path to xlsx}
- CSV data: {absolute paths to data/*.csv}
- Known mistakes: {absolute path to references/known_mistakes.md}
- Step spec: {absolute path to references/steps/step_{NN}_{name}.md}

## Report Output
{absolute path to {TICKER}_reports/step_{NN}_adversarial_audit.md}
""")
```

**WAIT. If adversarial finds issues:**
- If confidence = HIGH: spawn builder to fix, then re-audit
- If confidence = MEDIUM/LOW: spawn dcf-auditor as verification auditor to confirm/deny the finding
- Only verified-real issues go back to builder

#### E. COMMIT AND ADVANCE
Once both auditors PASS:
```bash
git add autodcf/{group}/step_{NN}_{name}.py {TICKER}_reports/step_{NN}_*.md
git commit -m "Step {N}: {Name} — built, checklist PASS, adversarial PASS"
```

Update BUILD_STATE.md: step status = passed, audit iterations recorded.

### Phase 3: Integration Audit (Step 18)

After all 17 build steps pass, spawn ONE comprehensive integration audit + ONE adversarial integration audit. See `references/steps/step_18_integration_audit.md` for the full checklist.

## ENFORCEMENT (STRUCTURAL)

The orchestrator architecture makes batching **structurally impossible**:

1. **Each builder receives ONE step's spec.** It literally cannot build Step 6 because it only has Step 5's file paths.
2. **Each auditor receives ONE step's checklist.** It cannot audit Steps 5+6 together because the prompt only contains Step 5's checklist.
3. **The orchestrator controls the loop.** It spawns builder, waits, spawns auditor, waits, spawns adversarial, waits. There is no way to skip ahead.
4. **Report validation is automatic.** The orchestrator reads the checklist audit report and counts addressed items vs total items. Missing items = re-spawn.

**You (the orchestrator) NEVER write build code.** You ONLY:
- Read specs and state files
- Construct agent prompts with file paths
- Spawn agents and wait for results
- Validate audit report completeness
- Update BUILD_STATE.md
- Make git commits

## Resume Protocol

If conversation context resets mid-build:
1. Read `{TICKER}_BUILD_STATE.md` — exact step, status, which audits passed
2. Read the report index — which reports exist
3. Read the latest build report — what was last built
4. Verify xlsx state with openpyxl (quick sanity check)
5. Resume from the exact step/status recorded

## Completion Criteria

The model is DONE when:
- All 7 data sub-steps completed and validated (0g PASS)
- All 17 build steps individually audited (checklist + adversarial PASS)
- Step 18 integration audit passed (checklist + adversarial PASS)
- 7 Cover checks all evaluate to PASS
- UFCF reconciliation = 0
- Driver switch cascades end-to-end
- 8 sensitivity tables present with re-discounting formulas
- Cumulative scenario comparison present
- Every blue-font cell traceable to CSV row with source quote
- Zero Excel errors (#REF!, #DIV/0!, #VALUE!, #NAME?)
